On-chain KMS uses a smart contract on Ethereum or Base as the key management layer instead of the centralized Phala KMS. This provides decentralized access control — only the contract owner can authorize compose file changes and environment variable updates.

## PHALA vs On-Chain KMS

| Feature | PHALA KMS | On-Chain KMS (ETHEREUM/BASE) |
|---------|-----------|------------------------------|
| Key management | Centralized Phala service | Smart contract on-chain |
| App ID source | Returned by `provisionCvm` | Generated by `deployAppAuth` contract |
| Compose updates | Direct API call | Two-phase: register on-chain, then API call |
| Env var updates | Direct API call | Two-phase when `env_keys` change |
| Cost | Free | Gas fees for contract interactions |
| Chain | None | Ethereum mainnet or Base |

## Full Deploy Flow

### Step 1: Provision the CVM

```typescript
import { createClient, encryptEnvVars, parseEnvVars } from "@phala/cloud";
import { deployAppAuth } from "@phala/cloud";

const client = createClient();

const provision = await client.provisionCvm({
  name: "my-app",
  compose_file: {
    docker_compose_file: composeYaml,
    allowed_envs: ["API_KEY"],
  },
  kms: "ETHEREUM", // or "BASE"
});
```

With on-chain KMS, `provision.app_id` is `null` — you need to deploy a contract to get one.

### Step 2: Fetch KMS Details

```typescript
const kmsList = await client.getKmsList({ is_onchain: true });
const kms = kmsList.items.find(k => k.id === provision.kms_id || k.slug === provision.kms_id);

// kms.chain — viem Chain config
// kms.kms_contract_address — factory contract address
```

### Step 3: Deploy AppAuth Contract

```typescript
const deployed = await deployAppAuth({
  chain: kms.chain,
  rpcUrl: "https://eth-mainnet.g.alchemy.com/v2/...",
  kmsContractAddress: kms.kms_contract_address,
  privateKey: "0xabc123...",
  deviceId: provision.device_id,
  composeHash: provision.compose_hash,
});

console.log(`App ID: ${deployed.appId}`);
console.log(`Contract: ${deployed.appAuthAddress}`);
```

### Step 4: Encrypt Environment Variables

```typescript
// Get the encryption public key from KMS
const { public_key } = await client.getAppEnvEncryptPubKey({
  kms: kms.slug,
  app_id: deployed.appId,
});

const envVars = parseEnvVars("API_KEY=my-secret-key");
const encrypted = await encryptEnvVars(envVars, public_key);
```

### Step 5: Commit the Provision

```typescript
const cvm = await client.commitCvmProvision({
  app_id: deployed.appId,
  compose_hash: provision.compose_hash,
  encrypted_env: encrypted,
  env_keys: ["API_KEY"],
  kms_id: kms.slug,
  contract_address: deployed.appAuthAddress,
  deployer_address: deployed.deployer,
});

console.log("CVM deployed:", cvm.id);
```

## Updating Environment Variables

When you update env vars on an on-chain KMS CVM and the set of `env_keys` changes, the API returns `precondition_required`:

```typescript
import { addComposeHash } from "@phala/cloud";

// Phase 1: attempt the update
const result = await client.updateCvmEnvs({
  id: "my-app",
  encrypted_env: newEncrypted,
  env_keys: ["API_KEY", "NEW_VAR"], // env_keys changed
});

if (result.status === "precondition_required") {
  // Phase 2a: register the new compose hash on-chain
  const receipt = await addComposeHash({
    chain: result.kms_info.chain,
    kmsContractAddress: result.kms_info.kms_contract_address,
    appId: result.app_id as `0x${string}`,
    composeHash: result.compose_hash,
    privateKey: privateKey,
  });

  // Phase 2b: retry with on-chain proof
  await client.updateCvmEnvs({
    id: "my-app",
    encrypted_env: newEncrypted,
    env_keys: ["API_KEY", "NEW_VAR"],
    compose_hash: result.compose_hash,
    transaction_hash: receipt.transactionHash,
  });
}
```

If the env keys don't change (same keys, different values), the update goes through in a single call.

## Updating Docker Compose

The same two-phase flow applies to Docker Compose updates:

```typescript
// Phase 1
const result = await client.updateDockerCompose({
  id: "my-app",
  docker_compose_file: newYaml,
});

if (result.status === "precondition_required") {
  // Register on-chain
  const receipt = await addComposeHash({
    chain: result.kms_info.chain,
    kmsContractAddress: result.kms_info.kms_contract_address,
    appId: result.app_id as `0x${string}`,
    composeHash: result.compose_hash,
    privateKey: privateKey,
  });

  // Retry with proof
  await client.updateDockerCompose({
    id: "my-app",
    docker_compose_file: newYaml,
    compose_hash: result.compose_hash,
    transaction_hash: receipt.transactionHash,
  });
}
```

## Updating via Compose File API

For more control, use the two-phase compose file update API:

```typescript
// Get current compose file
const compose = await client.getCvmComposeFile({ id: "my-app" });
compose.docker_compose_file = newYaml;
compose.allowed_envs = ["API_KEY", "NEW_VAR"];

// Provision the update (get compose_hash)
const provision = await client.provisionCvmComposeFileUpdate({
  uuid: "my-app",
  app_compose: compose,
});

// Register on-chain
const receipt = await addComposeHash({
  chain: cvm.kms_info.chain,
  rpcUrl: rpcUrl,
  appId: cvm.app_id as `0x${string}`,
  composeHash: provision.compose_hash,
  privateKey: privateKey,
});

// Commit the update
await client.commitCvmComposeFileUpdate({
  id: "my-app",
  compose_hash: provision.compose_hash,
  encrypted_env: encrypted,
  env_keys: ["API_KEY", "NEW_VAR"],
});
```

## Device Management

On-chain KMS uses a device allowlist in the smart contract. These functions interact directly with the blockchain to manage which devices can run your app.

### addDevice

Adds a device to the DstackApp contract's allowlist. This is an on-chain transaction that requires ETH for gas.

**Parameters:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `chain` | `Chain` | Conditional | Viem chain config (required if no `walletClient`) |
| `appAddress` | `Address` | Yes | DstackApp contract address |
| `deviceId` | `string` | Yes | Device ID to add (hex) |
| `privateKey` | `Hex` | Conditional | Caller private key (provide this or `walletClient`) |
| `walletClient` | `WalletClient` | Conditional | Custom wallet client |
| `rpcUrl` | `string` | No | Custom RPC URL |
| `publicClient` | `PublicClient` | No | Custom public client |
| `skipPrerequisiteChecks` | `boolean` | No | Skip balance checks (default: `false`) |
| `timeout` | `number` | No | Transaction timeout in ms (default: 120000) |

**Returns:** `AddDevice` — `{ appAddress, deviceId, transactionHash, blockNumber?, gasUsed? }`

```typescript
import { addDevice } from "@phala/cloud";
import { base } from "viem/chains";

const result = await addDevice({
  chain: base,
  appAddress: "0x1234...abcd",
  deviceId: "0xaabbccdd...",
  privateKey: "0x...",
});
console.log(`Device added: ${result.transactionHash}`);
```

### setAllowAnyDevice

Toggles the `allowAnyDevice` flag on the DstackApp contract. When enabled, the device allowlist is bypassed and any device can run the app.

**Parameters:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `chain` | `Chain` | Conditional | Viem chain config (required if no `walletClient`) |
| `appAddress` | `Address` | Yes | DstackApp contract address |
| `allow` | `boolean` | Yes | Whether to allow any device |
| `privateKey` | `Hex` | Conditional | Caller private key (provide this or `walletClient`) |
| `walletClient` | `WalletClient` | Conditional | Custom wallet client |
| `rpcUrl` | `string` | No | Custom RPC URL |
| `timeout` | `number` | No | Transaction timeout in ms (default: 120000) |

**Returns:** `SetAllowAnyDevice` — `{ appAddress, allow, transactionHash, blockNumber?, gasUsed? }`

```typescript
import { setAllowAnyDevice } from "@phala/cloud";

await setAllowAnyDevice({
  chain: base,
  appAddress: "0x1234...abcd",
  allow: true,
  privateKey: "0x...",
});
```

### getAllowedDevices

Queries the on-chain device allowlist for a DstackApp contract. Checks a list of candidate device IDs and returns which ones are actually allowed. This is a read-only operation that does not require gas.

**Parameters:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `chain` | `Chain` | Conditional | Viem chain config (required if no `publicClient`) |
| `appAddress` | `Address` | Yes | DstackApp contract address |
| `deviceIds` | `string[]` | Yes | Candidate device IDs to check |
| `rpcUrl` | `string` | No | Custom RPC URL |
| `publicClient` | `PublicClient` | No | Custom public client |

**Returns:** `GetAllowedDevices`

| Field | Type | Description |
|-------|------|-------------|
| `appAddress` | `string` | Contract address |
| `owner` | `string` | Contract owner address |
| `allowAnyDevice` | `boolean` | Whether any device is allowed |
| `devices` | `string[]` | Device IDs that are allowed (from candidates) |

```typescript
import { getAllowedDevices } from "@phala/cloud";

const result = await getAllowedDevices({
  chain: base,
  appAddress: "0x1234...abcd",
  deviceIds: ["0xaa...", "0xbb..."],
});
console.log(`Allowed devices: ${result.devices}`);
console.log(`Allow any: ${result.allowAnyDevice}`);
```

### checkDeviceAllowed

Checks whether a specific device is allowed by a DstackApp contract. Reads the `allowAnyDevice()` flag first, then checks the specific device ID. This is a read-only operation.

**Parameters:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `chain` | `Chain` | Conditional | Viem chain config (required if no `publicClient`) |
| `appAddress` | `Address` | Yes | DstackApp contract address |
| `deviceId` | `string` | Yes | Device ID to check |
| `rpcUrl` | `string` | No | Custom RPC URL |
| `publicClient` | `PublicClient` | No | Custom public client |

**Returns:** `boolean` — whether the device is allowed.

```typescript
import { checkDeviceAllowed } from "@phala/cloud";

const allowed = await checkDeviceAllowed({
  chain: base,
  appAddress: "0x1234...abcd",
  deviceId: "0xaabbccdd...",
```

### checkOnChainPrerequisites

Checks all on-chain prerequisites for a CVM update in a single RPC call. Uses multicall to batch-check the `allowAnyDevice` flag, device ID registration, and compose hash registration. This is the most efficient way to verify on-chain state before attempting an update.

**Parameters:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `chain` | `Chain` | Conditional | Viem chain config (required if no `publicClient`) |
| `appAddress` | `Address` | Yes | DstackApp contract address |
| `deviceId` | `string` | Yes | Device ID to check |
| `composeHash` | `string` | Yes | Compose hash to check |
| `rpcUrl` | `string` | No | Custom RPC URL |
| `publicClient` | `PublicClient` | No | Custom public client |

**Returns:** `OnChainPrerequisites`

| Field | Type | Description |
|-------|------|-------------|
| `allowAnyDevice` | `boolean` | Whether the contract allows any device |
| `deviceAllowed` | `boolean` | Whether the device is allowed (true if `allowAnyDevice` or specifically registered) |
| `composeHashAllowed` | `boolean` | Whether the compose hash is registered |

```typescript
import { checkOnChainPrerequisites } from "@phala/cloud";

const prereqs = await checkOnChainPrerequisites({
  chain: base,
  appAddress: "0x1234...abcd",
  deviceId: "0xaabbccdd...",
  composeHash: "0xeeff0011...",
});

if (!prereqs.deviceAllowed) {
  // Need to call addDevice first
}
if (!prereqs.composeHashAllowed) {
  // Need to call addComposeHash first
}
```
