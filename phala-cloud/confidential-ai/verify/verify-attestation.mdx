## Why attestation matters

Attestation proves your AI workloads run on genuine TEE hardware with verified software. It gives you confidence that:

- Hardware comes from trusted vendors (NVIDIA, Intel)
- The entire software stack is authentic and untampered
- Your workloads are protected from external attacks

## Verify TEE hardware stack

Hardware verification confirms your GPUs and CPUs are genuine and secure. This prevents counterfeit hardware from compromising your AI workloads.

You'll verify:
- **GPU authenticity** - NVIDIA GPUs are genuine and running in TEE mode
- **CPU authenticity** - Intel CPUs support TEE protections
- **Cryptographic security** - Hardware security features are operational

Follow this [Python example](https://github.com/Phala-Network/confidential-ai-verifier/blob/f606cf8675b6d920cdec49f46c5624cbcd7f2f16/examples/attestation_verifier.py) to verify the full hardware stack.

### Get the attestation report

Fetch the attestation report with a simple [GET request](https://github.com/Phala-Network/confidential-ai-verifier/blob/f606cf8675b6d920cdec49f46c5624cbcd7f2f16/examples/attestation_verifier.py#L31-L38). The report contains:

- **GPU attestation** - NVIDIA hardware verification data
- **CPU attestation** - Intel TEE verification data
- **Signing address** - Public key for signature verification

Start by verifying GPU and CPU attestation.

### Verify NVIDIA GPU attestation

Verify NVIDIA GPUs by sending the `nvidia_payload` to NVIDIA's attestation service:

1. [Send payload for GPU verification](https://github.com/Phala-Network/confidential-ai-verifier/blob/f606cf8675b6d920cdec49f46c5624cbcd7f2f16/examples/attestation_verifier.py#L43)
2. [Decode NVIDIA tokens](https://github.com/Phala-Network/confidential-ai-verifier/blob/f606cf8675b6d920cdec49f46c5624cbcd7f2f16/examples/attestation_verifier.py#L149-L168) - one token per GPU to validate each device

### Verify Intel TDX CPU attestation

Verify Intel CPUs using the [Intel quote](https://github.com/Phala-Network/confidential-ai-verifier/blob/f606cf8675b6d920cdec49f46c5624cbcd7f2f16/examples/attestation_verifier.py#L72):

1. Copy the `intel_quote` value
2. Paste it into the [TEE Attestation Explorer](https://proof.t16z.com/)
3. Confirm CPU authenticity through the verification tool

## Verify TEE software stack

With hardware verified, confirm the software stack is authentic and untampered. This ensures:

- No unauthorized modifications to OS or application code
- The TEE operates in a trusted security state
- The trust chain from hardware to application remains intact

Verify these software components:

### 1. Verify operating system integrity

Confirm the TEE runs the expected, unmodified OS using [this verification process](https://github.com/Phala-Network/dstack-verifier/blob/95689c41/src/verification/osVerification.ts#L13-L27):

1. **Measure OS image** - Calculate cryptographic hashes of OS components
2. **Compare with expected values** - Match against TCB (Trusted Computing Base) values
3. **Validate registers** - Ensure all system registers are correct


### 2. Verify source code authenticity

Verify the application code is unmodified by checking Docker Compose configuration hashes:

1. [Calculate compose hash](https://github.com/Phala-Network/dstack-verifier/blob/95689c41/src/dataObjects/baseDataObjectGenerator.ts#L341) - Compute SHA256 of Docker Compose config
2. [Compare with registry](https://github.com/Phala-Network/dstack-verifier/blob/95689c41/src/utils/dstackContract.ts#L52) - Check against allowed hashes in blockchain smart contract
3. [Validate event log](https://github.com/Phala-Network/dstack-verifier/blob/95689c41/src/dataObjects/baseDataObjectGenerator.ts#L341) - Cross-reference with deployment records

### 3. Verify distributed root-of-trust

The distributed root-of-trust (KMS) manages cryptographic keys and authenticates applications. It validates:

- Hardware attestation
- Operating system integrity
- Source code authenticity

See [KMS Verifier code](https://github.com/Phala-Network/dstack-verifier/blob/95689c415cad165033c700979cd26225c56c8d1c/src/verifiers/kmsVerifier.ts#L27) for implementation details.


### 4. Verify network end-to-end encryption

Ensure TEE controls all communication with [network verification](https://github.com/Phala-Network/dstack-verifier/blob/7d2f5acc04e498dced91334a1b5424040b44535e/src/verifiers/gatewayVerifier.ts#L40):

1. **TEE-controlled keys** - Verify keys are generated and stored in TEE
2. **Certificate matching** - Ensure TLS certificates match TEE public keys
3. **DNS CAA records** - Validate Certificate Authority Authorization
4. **Certificate transparency** - Analyze historical certificate patterns
